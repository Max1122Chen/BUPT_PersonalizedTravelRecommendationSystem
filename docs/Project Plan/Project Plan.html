项目愿景与业务目标

愿景

做出一个真正能跑起来、能演示、能解释清楚算法思路的旅游推荐系统，覆盖旅游前、中、后的关键场景。

业务目标

  1. 在旅游前提供可解释的目的地推荐与详情查询。
  2. 在旅游中提供可计算、可视化的最优路径规划和设施/美食服务。
  3. 在旅游后提供日记创作、浏览、检索与交流能力。
  4. 提供管理员可操作的数据管理能力，支撑内容持续更新。

项目成功标准

  1. 关键业务链路端到端可演示，且核心场景无阻断缺陷。
  2. 核心需求实现率达到 90% 以上（以需求清单计）。
  3. 关键性能指标达到目标范围：接口响应时间、路径规划计算时间、检索响应时间。
  4. 关键数据结构（MyList/MyMap/MySet/MyPriorityQueue）在核心算法模块完成替换并可验证。
  5. 交付文档、代码和验证记录之间保持一致且可追溯。

暂缓需求

  1. 日记压缩存储。
  2. AIGC 旅游动画。
  3. 完整数据备份与恢复自动化。

需求边界定义

  1. 本期聚焦核心业务闭环，不追求一次性覆盖所有低优先级创新功能。
  2. 对暂缓需求保留接口扩展位与数据结构扩展位，避免后续重构成本过高。

技术方案与技术栈

总体架构

采用前后端分离架构：前端负责交互与可视化，后端负责业务规则、算法与安全控制，数据库负责持久化。

技术栈总览

层级 技术 版本/说明 用途
前端 Vue 3 + TypeScript + Vite 现有工程基线 页面与交互实现
前端 Pinia + Vue Router + Axios 现有工程基线 状态管理、路由与接口通信
前端 Element Plus + ECharts 现有工程基线 UI 组件与图形可视化
后端 Java + Spring Boot 现有工程基线 API 与业务服务
后端 MyBatis-Plus 现有工程基线 数据访问层
数据库 MySQL 现有工程基线 持久化存储
缓存 Redis（可选） 预留能力 热点数据与扩展优化

核心实现约束

  1. 数据库仅承担持久化职责。
  2. 运行时采用数据预加载后在内存中进行主要计算。
  3. 搜索/模糊/全文检索/关联匹配不使用 SQL LIKE/REGEXP/FULLTEXT/JOIN 作为计算实现。
  4. 核心数据结构与算法模块采用自定义结构实现，避免直接依赖 Java 内置集合实现作为最终方案。
  5. 所有接口统一响应结构为 code/data/message。

数据结构实现优先级（前置）

  1. 优先级 P0：先完成 MyList/MyMap/MySet/MyPriorityQueue 基础结构与单元验证。
  2. 优先级 P0：在路线规划、Top-K、检索索引三个核心模块中优先替换。
  3. 优先级 P1：在服务层非核心路径逐步替换。
  4. 验收口径：核心算法模块不再直接依赖 java.util 集合作为最终实现。

推荐模块

  1. 完成兴趣权重建模与推荐打分。

  3. 完成美食详情与评分能力。

日记模块

  1. 完成日记创建、编辑、删除、详情与列表。
  2. 完成目的地关联检索、名称检索、内容检索。
  3. 完成日记评分、热度计算与推荐排序。

管理模块

  1. 完成景区、建筑、道路、美食基础管理能力。

质量计划与测试策略

质量目标

  1. 功能质量：关键业务用例通过率不低于 95%。
  2. 稳定性：核心链路无阻断级缺陷。
  3. 性能：满足需求文档给出的时间指标。
  4. 合规性：满足核心技术约束和接口规范。

测试分层

  1. 单元测试：覆盖路线、推荐、检索、日记核心服务。
  2. 接口测试：覆盖鉴权、参数校验、异常处理、统一响应结构。
  3. 集成测试：覆盖登录、推荐、路径规划、日记发布完整链路。
  4. 回归测试：每次里程碑前执行关键场景回归。

质量门禁

  1. 未通过核心回归测试不得进入封版分支。
  2. 不满足约束扫描项不得合并核心模块变更。
  3. 文档未同步更新不得关闭需求任务。

AI辅助开发方案

定位说明

AI 在本项目中是全栈协作角色，覆盖需求分析、技术设计、编码实现、测试与文档维护；但最终责任仍由项目成员承担。

全流程参与矩阵

阶段 人类主责 AI 参与内容 输出物
需求阶段 确认需求边界与优先级 需求拆解、冲突检查、验收标准草案 需求分解表、验收清单初稿
设计阶段 架构决策与技术定稿 技术选型对比、接口草案、风险清单建议 设计草案、接口草案、风险台账
开发阶段 核心实现与代码审查 代码生成、重构建议、缺陷定位、边界条件补全 模块实现、变更说明、代码评审记录
测试阶段 测试决策与结果确认 测试用例生成、回归点建议、异常路径构造 测试清单、回归报告
交付阶段 最终验收与答辩 文档一致性检查、交接记录整理、答辩问答辅助 交付包、交接日志、答辩提纲

AI 驱动开发流程

  1. 会话启动：先加载项目上下文与当前任务边界（需求范围、影响模块、验收标准）。
  2. 方案阶段：AI 先输出“实现路径 + 风险点 + 验证方式”。
  3. 实施阶段：按“小步修改 -> 即时检查 -> 记录变更”推进。
  4. 验证阶段：AI 生成测试建议并执行核心链路验证。
  5. 收尾阶段：更新交接日志，补充文档变更与风险状态。

各阶段AI操作清单

阶段 AI必做动作 人工确认点
需求阶段 输出需求拆解、边界冲突、验收草案 确认最终范围与优先级
设计阶段 输出接口草案、模块依赖、风险清单 定稿技术方案与接口契约
开发阶段 生成代码草稿、重构建议、约束检查 审查核心算法实现与正确性
测试阶段 生成测试用例与回归点 确认测试结果与缺陷定级
交付阶段 汇总变更、整理文档、生成答辩提纲 最终验收与签字归档

质量与治理机制

  1. 所有 AI 代码输出必须人工审查后合入。
  2. 核心算法与约束相关改动必须附验证证据（复杂度说明/样例结果/回归记录）。
  3. AI 生成内容必须遵守项目硬约束（数据库职责、内存检索、自定义数据结构）。
  4. 每次会话必须同步更新交接日志，确保跨会话连续性。

AI 使用效果度量（项目管理视角）

  1. 需求处理效率：需求拆解到任务可执行的平均耗时。
  2. 开发效率：同类模块的实现与联调耗时变化。
  3. 缺陷质量：回归缺陷数与阻断缺陷修复时长。
  4. 文档质量：需求、设计、代码、测试、交接的一致性通过率。

AI相关风险与控制

  1. 风险：AI建议偏离约束。

控制：每次提交前执行约束检查清单（SQL检索、内置集合、响应结构）。

  1. 风险：AI生成代码可运行但不可维护。

控制：强制代码评审，新增复杂逻辑必须附注释与测试依据。

  1. 风险：跨会话上下文断裂。

控制：会话开始先读取 PROJECT_CONTEXT、WORKFLOW、HANDOFF。

配置管理与交付管理

代码与分支策略

  1. 采用主干 + 功能分支模式。
  2. 每次提交保持单一逻辑变更，提交信息可追踪。
  3. 敏感配置使用环境变量，不在仓库中明文暴露。

文档管理

  1. 需求、设计、计划、交接四类文档必须同步维护。
  2. 每个里程碑结束执行文档一致性核查。

交付物清单

  1. 可运行前后端系统。
  2. 需求文档、技术设计文档、开发规范、项目计划书。
  3. 测试记录、缺陷清单、风险台账、演示脚本。

格式迁移规范（Markdown -> HTML -> Word）

  1. 标题不超过三级，确保目录可自动生成。
  2. 表格控制在 6 列以内，避免 Word 跨页断裂。
  3. 列表采用单层编号，避免复杂嵌套。
  4. 每个章节固定使用“目标-方法-输出”结构，便于模板化转换。
  5. 图表采用“图号、标题、说明”三段式，方便粘贴后统一样式。
